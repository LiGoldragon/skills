# Skill — context maintenance

## Drain accumulated context into durable homes

Context maintenance turns lane artifacts into smaller durable guidance. Preserve useful substance; retire transcript, correction history, duplicate summaries, and stale working notes.

Use the existing authority order:

- psyche intent goes through the accepted intent path;
- repository operating guidance goes to `AGENTS.md`;
- architecture facts go to `ARCHITECTURE.md`;
- user-facing or project overview material goes to `README.md`;
- active work items go to the tracker;
- obsolete artifacts are superseded or deleted after their useful content moves.

## Classify each artifact

Each item has one fate:

- **Keep** when it remains an active pickup surface.
- **Migrate** when it contains durable substance that belongs in a repo file, skill, tracker item, or intent record.
- **Retire** when it is transcript, outdated planning, duplicate evidence, or superseded notes.

Do not preserve an artifact merely because it exists. Do not delete before moving the load-bearing facts.

## Inventory narrowly

Start with the lane or directory named by the task. List files, owners, recency, and whether another agent may still need them. Avoid broad workspace sweeps unless the task explicitly asks for them.

## Migrate by content type

Move operational rules into the nearest owning source. Move design constraints into architecture. Move code-facing tasks into tracker items with completion evidence. Move reusable agent discipline into skills only when it is not repo-specific.

When competing alternatives exist, keep the accepted decision and the reason. Delete or supersede rejected alternatives after any still-useful constraints are moved.

## Retire safely

Before deleting or superseding an artifact, verify that no active manifest, tracker item, prompt, or handoff references it as the pickup surface. If references remain, update them or leave a clear supersession pointer.

Prefer deletion for dead notes. Keep a short rollover only when another fresh agent needs an active pickup point.

## Use helpers only for bounded reading

A helper may inventory or summarize a bounded pile, but the lead owns disposition and writes durable changes. Read helper outputs before acting; treat their recommendations as evidence, not authority.

## Anti-patterns

- turning maintenance into a new report pile;
- copying transcript into durable files;
- restating skill doctrine in handoffs;
- storing repo-specific policy in a shared skill;
- deleting artifacts whose substance has not moved;
- preserving all uncertainty instead of naming the decision that remains open.
