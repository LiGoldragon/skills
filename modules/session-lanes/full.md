# Skill — session lanes

## Rules

A discipline is the persistent agent identity: skill set, authority class, memory surface, and signing key. A lane is one work session named for its intent. Do not use a discipline name as the lane name.

A lane carries its discipline as metadata. The discipline says what kind of agent acts; the lane says what this session is for.

Use the lane name supplied by the harness or psyche. Do not infer it from a neighboring session, role packet, directory name, or prior convention. If the lane is missing and the task needs durable artifacts, derive one stable CamelCase name from the work title.

Current Orchestrate coordination names are an interim compatibility surface, not full lanes. For editing claims, use the dispatcher-assigned unique coordination name; if it is missing, pause instead of deriving a claim owner from the role or discipline.

## Registration

When a live lane registry is in scope, register through the owner-only meta CLI, not the working CLI. The role vector ends with the base discipline and preceding tokens name the session intent, for example `[NewLanesDesign Designer]`.

Observe active lanes through the working registry surface when coordination depends on whether a lane exists.

## Lifecycle

Prefer a fresh lane over endless context compaction. Use the early high-fidelity window for alignment and design. Delegate bounded reading or mechanical exploration instead of spending the lane's deep context on broad inspection.

At drain, route every leftover idea to exactly one fate:

- accepted durable intent;
- tracked work;
- owned source documentation;
- abandoned as landed, stale, or wrong.

A lane is retired only after no active pickup surface still depends on its leftover context.
