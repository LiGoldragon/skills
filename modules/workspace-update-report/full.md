# Skill — workspace update report

## Rules

Write a workspace update only when the psyche asks for one, a long busy period
needs synthesis, or a context-loss event needs a durable direction summary.

The update explains direction, not chronology. It says what changed, why the
change matters, what evidence supports it, and what work now depends on it.

Pick a clear baseline and state it. Prefer the prior update report; otherwise
choose the last stable checkpoint that makes the period intelligible.

Survey repository history, changed durable artifacts, landed reports, retired
surfaces, generated doctrine changes, and component state. Skim broadly first;
read deeply only where the movement changed direction.

Do not narrate every commit, session, tool call, or file. Group motion by theme
and cite only the artifacts that prove the theme.

Separate observed changes from interpretations. A changed file, accepted record,
merged report, or passing check is observation. A trend, risk, or next emphasis
is interpretation.

Name the highest-signal shifts first. Each headline needs a concrete source and
a plain-language consequence.

When durable human intent changed, quote the load-bearing wording sparingly and
explain how it affected work. Do not duplicate the intent corpus.

When skills, repository instructions, or architecture documents changed, explain
the operational effect rather than restating the new text.

When reports landed or retired, state what decision or evidence they preserve and
whether later work should read them.

When code moved, identify the component, behavior, checks, and remaining risk.
Avoid implementation minutiae unless it changes how agents should work.

Keep forward-looking notes grounded in already-open work, blockers, or explicit
psyche decisions. Do not invent a roadmap.

The finished update should be short enough to read in one sitting. Delete any
paragraph that does not help the psyche understand the period's direction.
